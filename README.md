# RskMedia

Самохостируемая медиа-галерея с хранением файлов в S3, спроектированная для быстрого просмотра с телефонов и десктопных браузеров через локальную Wi-Fi сеть или публичный интернет.

Приложение отдаёт фото и видео через локальный прокси, поддерживает потоковое чтение с HTTP Range, и поставляется с системой авторизации по токенам, админ-панелью для управления контентом, поиском дубликатов по перцептивному хешу и JSON-бэкапами базы данных.

## Обзор

RskMedia — это приложение на Node.js / Express, которое хранит бинарные медиафайлы в S3-совместимом бакете, а реляционные метаданные (категории, подкатегории, возрастные рейтинги, избранное, перцептивные хеши, токены) — в PostgreSQL.

Фронтенд — ванильный JavaScript с собственными CSS-стилями, без React, без бандлера, без шага сборки. Один и тот же код обслуживает галерею, страницу избранного, страницу логина и админ-панель.

Приложение построено вокруг четырёх идей:

1. **Локальный доступ к медиа** — файлы отдаются через прокси `/media/:type/:id` на том же Node-сервере, что обходит проблемы файрволов и CORS, когда клиенты (телефоны) подключены к Wi-Fi сетям, отличным от сети S3-эндпоинта.
2. **Потоковое чтение по HTTP Range** — видео и большие изображения стримятся как `206 Partial Content`, поэтому перемотка в браузерном плеере работает без скачивания всего файла.
3. **Контроль доступа по токенам** — паролей нет. Администратор выдаёт непрозрачные токены формата `client_<32hex>` или `admin_<32hex>`; токен хешируется bcrypt на сервере, выданный JWT тоже хешируется bcrypt на сервере, а логаут инвалидирует сессию очисткой хеша JWT.
4. **Операционные инструменты** — админ-панель включает бэкап/восстановление, поиск дубликатов, пакетное редактирование, статистику и управление жизненным циклом токенов.

## Возможности

### Галерея

- Бесконечная прокрутка с фильтрами, синхронизированными с URL (категория, подкатегория, возрастной рейтинг, сортировка)
- Поиск по ключу медиа, названию категории или подкатегории
- Полноэкранный просмотрщик с навигацией с клавиатуры (`←`, `→`, `Esc`), свайпом на тачскрине и pinch-zoom + drag-pan для изображений
- Прогрессивный апгрейд изображений — миниатюры заменяются на 1920 px вариант когда карточка попадает в область видимости (с плавным переходом opacity, чтобы избежать мерцания)
- Авто-скрытие элементов управления в просмотрщике через 3 секунды бездействия мыши/тача/клавиатуры
- Избранное — кнопка-сердце на каждой карточке и в модалке, batch-check API для сетки, отдельная страница `/favorites`
- Бейдж возрастного рейтинга на каждой карточке и в просмотрщике

### Админ-панель

Семь вкладок, все за ролью `admin_`:

| Вкладка | Что делает |
|---------|------------|
| **Загрузка** | Одиночная загрузка с прогресс-баром через XHR, пакетная загрузка drag-and-drop с лимитом параллельности, прогрессом по каждому файлу и кнопкой отмены |
| **Редактирование** | Бесконечная прокрутка всех медиа с inline-выпадающими списками категории/подкатегории/возраста; массовое выделение с применением или удалением |
| **Категории** | Создание, переименование и удаление категорий и подкатегорий; отказывается удалять категорию, в которой ещё есть медиа или подкатегории |
| **Токены** | Создание новых токенов (client или admin), переключение активности, установка срока действия, удаление (нельзя удалить свою собственную сессию) |
| **Бэкап** | Скачивание JSON-снимка всех пяти таблиц (токены экспортируются без поля `jwt_hash`); восстановление из ранее скачанного JSON со структурной и типовой валидацией |
| **Дубликаты** | Запуск сканирования перцептивного хеша (64-bit dHash через `sharp`, LSH-бакетирование, расстояние Хэмминга ≤ 10) и визуальный осмотр групп дубликатов |
| **Статистика** | Шесть агрегатных таблиц: разбивка по типу, распределение по возрасту, количество по категориям/подкатегориям, отчёт о пропущенных метаданных, недавние загрузки (24 ч / 7 дн / 30 дн) |

### Медиа-прокси

- `GET /media/:type/:id` и `HEAD /media/:type/:id`, где `type` — это `thumb`, `display` или `original`
- Полная поддержка HTTP Range — `bytes=START-END`, `bytes=START-`, `bytes=-SUFFIX`, возвращает `416` для невалидных диапазонов
- In-memory LRU-кеш для файлов ≤ 5 MB (≤ 500 записей, TTL 1 час)
- Дедупликация параллельных запросов — параллельные запросы одного и того же ключа делят одну загрузку из S3
- HEAD-запросы возвращают только метаданные, без тела
- Большие файлы стримятся напрямую из S3 без буферизации
- Кешированный контент отдаётся из RAM через `Buffer.slice` за субмиллисекундные времена

### Операционные

- Инициализация базы с идемпотентными миграциями (`scripts/init-db.js`)
- Первоначальный бутстрап: создаёт токен админа и однократно печатает его plaintext в консоль
- Скрипт бэкапа для устаревшей колонки `file_size` (`scripts/populate-file-sizes.js`)

## Технологический стек

### Бэкенд

| Слой | Технология |
|------|------------|
| Среда выполнения | Node.js ≥ 18 |

## Архитектура

```
                ┌─────────────────────────────────────────────┐
                │               Браузер                       │
                │  (галерея, просмотрщик, админ-панель)       │
                └────────────┬────────────────────────────────┘
                             │ HTTPS / HTTP
                             │
                ┌────────────▼────────────────────────────────┐
                │            Express app                      │
                │  ┌──────────────────────────────────────┐   │
                │  │  Helmet (CSP с nonce)               │   │
                │  │  CORS · Cookie parser · Compression  │   │
                │  │  CSRF-middleware                     │   │
                │  │  Rate limiters (api/upload/admin)    │   │
                │  └──────────────────────────────────────┘   │
                │                                             │
                │  /api/auth     /api/media     /api/admin    │
                │  /api/categories               /api/favorites
                │  /media/:type/:id  (Range-aware S3 прокси)  │
                └────────┬──────────────────────────┬─────────┘
                         │                          │
                  ┌──────▼──────┐            ┌──────▼──────┐
                  │ PostgreSQL  │            │  S3 бакет   │
                  │ (метаданные,│            │  (бинарные  │
                  │  токены,    │            │  медиа)     │
                  │  избранное) │            │             │
                  └─────────────┘            └─────────────┘
```

Сервер слушает на `0.0.0.0:3000`, поэтому один и тот же инстанс доступен и из LAN (телефоны в той же Wi-Fi), и из публичного интернета (за обратным прокси, если нужно). При старте сервер выводит в лог и loopback-URL, и LAN-IP.

## Быстрый старт

### Требования

- Node.js 18 или новее
- PostgreSQL 12 или новее
- S3-совместимый бакет (AWS S3, Beget Cloud, MinIO и т.п.)
- `ffmpeg` и `ffprobe` в `$PATH`, если планируете загружать видео

### 1. Клонирование и установка

```bash
git clone https://github.com/schwf3xlr/RskMedia.git
cd RskMedia
npm install
```

### 2. Конфигурация окружения

```bash
cp .env.example .env
# Отредактируйте .env — см. раздел «Конфигурация» ниже
```

### 3. Инициализация базы данных

```bash
npm run init-db
```

Команда создаёт схему, сидит дефолтные категории и **однократно печатает в stdout свежесгенерированный токен админа**. Сохраните его — это единственный раз, когда plaintext-токен показывается.

### 4. Запуск сервера

```bash
npm start          # продакшен
npm run dev        # nodemon, авто-перезапуск
```

### 5. Открыть приложение

Сервер печатает оба URL при старте:

```
Local:   http://localhost:3000
Network: http://192.168.x.x:3000
```

Перейдите по локальному URL, нажмите на ссылку логина и вставьте токен админа, который вы сохранили на шаге 3.

## Схема базы данных

Пять таблиц, все определены в `scripts/init-db.js`. Внешние ключи каскадны на удалении токенов и медиа, и `SET NULL` на удалении категорий/подкатегорий.

### `tokens`

| Колонка | Тип | Заметки |
|---------|-----|---------|
| `id` | `SERIAL PK` | |
| `token_hash` | `VARCHAR(255) UNIQUE NOT NULL` | bcrypt(12) от plaintext-токена |
| `jwt_hash` | `VARCHAR(255)` | bcrypt(12) от выданного JWT — очистка инвалидирует сессию |
| `type` | `VARCHAR(10)` | `CHECK IN ('client', 'admin')` |
| `created_at` | `TIMESTAMP` | `DEFAULT NOW()` |
| `expires_at` | `TIMESTAMP` | nullable |
| `is_active` | `BOOLEAN` | `DEFAULT true` |

### `categories`

| Колонка | Тип | Заметки |
|---------|-----|---------|
| `id` | `SERIAL PK` | |
| `name` | `VARCHAR(100) UNIQUE NOT NULL` | |
| `created_at` | `TIMESTAMP` | |

### `subcategories`

| Колонка | Тип | Заметки |

## Модель безопасности

### Content Security Policy

Helmet сконфигурирован с `useDefaults: false` и собственноручно собранным CSP, который внедряет per-request 16-байтный nonce в `script-src` и `style-src`. Источники изображений и медиа разрешают `https:`, чтобы подписанные S3 URL (когда не используется прокси) работали без нарушения политики.

Стандартный бандл дефолтов намеренно отключён, чтобы Helmet не реинжектил `upgrade-insecure-requests` или `cross-origin-*` политики, которые сломали бы LAN-доступ по обычному HTTP.

### CSRF

Double-submit cookie pattern (`middleware/csrf.js`):

- Cookie `csrf_token` устанавливается в каждом ответе (подписанная, httpOnly=false на ответе, sameSite=lax, secure в продакшене).
- Фронтенд читает её из тега `<meta name="csrf-token">` в `<head>` и отправляет как `X-CSRF-Token` в каждом API-запросе.
- Middleware сравнивает значение подписанной cookie с заголовком (или полем тела `_csrf`) в каждом запросе, отличном от `GET`/`HEAD`/`OPTIONS`.

### Аутентификация

- Токены генерируются как `<type>_<32 hex chars>` и хешируются bcrypt (12 раундов) перед сохранением.
- При логине подписывается JWT секретом `JWT_SECRET` и сохраняется в подписанной httpOnly cookie.
- Bcrypt-хеш JWT пишется в `tokens.jwt_hash`. Каждый аутентифицированный запрос перепроверяет, что JWT-хеш всё ещё совпадает — так логаут может мгновенно инвалидировать сессии без ротации секрета cookie.
- `JWT_SECRET` и `COOKIE_SECRET` должны быть ≥ 32 символов; сервер завершается, если это не так.

### Авторизация

- `authenticateToken` заполняет `req.user` найденной строкой токена.
- `requireAdmin` проверяет `req.user.type === 'admin'` и возвращает 403 в противном случае.
- Все эндпоинты `/api/admin/*` требуют обе проверки.

### Rate-лимитирование

Четыре инстанса `express-rate-limit`, применяемых per-route:

| Лимитер | По умолчанию | Применяется к |
|---------|--------------|---------------|
| `apiLimiter` | 300 / окно | большинству `/api/*` |
| `uploadLimiter` | 50 / окно | эндпоинтам загрузки |
| `adminLimiter` | 200 / окно | `/api/admin/*` |
| `authLimiter` | (по умолчанию) | `/api/auth/*` |

### Валидация ввода

Цепочки `express-validator` определены inline в обработчиках роутов и извлекаются через `middleware/validate.js`. Загрузки дополнительно проходят через `helpers/fileValidator.js`, который проверяет magic bytes, чтобы отклонить спуфинг MIME-типов.

## Производительность

### Пайплайн изображений (при загрузке)

1. `sharp` читает буфер и зондирует метаданные
2. Генерируется миниатюра 400 px и загружается как `thumbnail_s3_key`
3. Если источник шире 1920 px, вариант 1920 px загружается как `display_s3_key`
4. Оригинал загружается как `s3_key`

### Прогрессивное отображение (при просмотре)

Галерея сразу отдаёт миниатюры и переключается на display-вариант только когда карточка попадает в область видимости (через `IntersectionObserver`). Подмена прячется за 310 мс переходом opacity, чтобы избежать мерцания.

### Кеш прокси

`routes/mediaProxy.js` держит небольшой in-memory LRU:

- Файлы ≤ 5 MB кешируются как `Buffer`
- Кеш вмещает до 500 записей, вытеснение через 1 час
- Параллельные запросы одного ключа делят одну in-flight загрузку
- Range-запросы обходят кеш и стримятся напрямую из S3

### Кеш signed URL

`config/s3.js` держит LRU presigned URL (по умолчанию: 5 000 записей, TTL 55 мин), чтобы избежать стоимости повторного подписания одинаковых URL на каждой загрузке страницы.

### Индексы

Все фильтруемые колонки проиндексированы, а самый частый паттерн выдачи (`category_id` + `age_rating` + `ORDER BY uploaded_at DESC`) покрыт композитным индексом.

## Разработка

### Скрипты

| Команда | Что делает |
|---------|------------|
| `npm start` | Запуск через `node app.js` |
| `npm run dev` | Запуск через `nodemon` — авто-перезапуск при изменении файлов |
| `npm run init-db` | Идемпотентно — создаёт схему, если её нет, прогоняет миграции, сидит категории, печатает бутстрап-токен админа |

### Стиль кода

- CommonJS на бэкенде (`require` / `module.exports`)
- ES-модули на фронтенде (`type="module"`)
- Без транспайлера — предполагается нативный браузерный ES2022
- Без бандлера — файлы отдаются напрямую из `public/`
- CSS разбит по зонам ответственности: переменные и reset в `base.css`, лейаут страниц в `layout.css`, переиспользуемые виджеты в `components.css`, специфичные для страниц переопределения в `pages.css`

### Логирование

`morgan('dev')` в разработке, `morgan('combined')` в продакшене. Ошибки логируются центральным обработчиком в `app.js` и возвращаются как JSON для запросов `/api/*` или рендерят `error.ejs` для страничных запросов (сообщение скрыто в продакшене).

## Заметки по деплою

- **Адрес прослушивания**: сервер биндится на `0.0.0.0`, поэтому принимает соединения на всех интерфейсах. За обратным прокси (nginx, Caddy) поставьте `TRUST_PROXY` под количество хопов, чтобы Express видел настоящий IP клиента.
- **TLS**: приложение само TLS не терминирует. Запускайте его за обратным прокси или на управляемой платформе, которая это делает.
- **`uploads/`**: временная папка Multer. Безопасно удалять между перезапусками; файлы стримятся в S3 во время запроса.
- **`NODE_ENV=production`**: включает secure-cookie, более длинное кеширование статики, `morgan('combined')` и скрывает сообщения об ошибках от конечных пользователей.
- **Service worker**: приложение намеренно отдаёт самоотключающийся service worker по `/sw.js`, чтобы обеспечить сброс клиентских кешей браузера после обновлений.
- **ffmpeg/ffprobe**: нужны на хосте, если планируете загружать видео. Сервер вызывает `fluent-ffmpeg` для валидации (`ffprobe`) и извлечения превью (на отметке 1 секунды).
- **Пример обратного прокси (Caddy)**:

```
rskmedia.example.com {
    reverse_proxy 127.0.0.1:3000
    encode gzip zstd
}
```
